# Part 4 — Apply appropriate visual smoothing

Some continuous time-series lines are visually noisy because every raw observation is joined with straight segments.

Use restrained Plotly spline interpolation where appropriate.

Suggested style:

```python
line={
    "color": color,
    "width": 2.2,
    "shape": "spline",
    "smoothing": 0.5,
}
```

Use smoothing conservatively.

Suitable continuous series include:

* CPI;
* core CPI;
* housing-cost inflation;
* unemployment;
* employment;
* wage growth;
* money growth;
* mortgage rates;
* house-price growth;
* regional rebased indices;
* normalised equity series;
* currency indices;
* commodity-price series.

Do not use spline smoothing for:

* Bank Rate;
* policy step series;
* bars;
* quarterly GDP bars;
* transaction volumes;
* lending-flow bars;
* sparse event series;
* any series where curved interpolation would imply misleading intermediate values.

Bank Rate should remain:

```python
shape="hv"
```

Do not apply statistical rolling averages unless a chart explicitly requires one.

The objective is visual interpolation, not alteration of the underlying observations.

# Part 14 — Redesign the Sector Proxy Performance chart

The current “Sector Proxy Performance” chart is not useful because one series dominates the scale.

Current behaviour:

* Sage has appreciated by tens of thousands of percent on the rebased scale;
* Barclays, BP, Rio Tinto and GSK are visually compressed near the x-axis;
* the chart therefore fails to communicate comparative performance;
* the y-axis is technically correct but practically unreadable.

Do not leave the chart on a single linear scale in its current form.

Investigate the most appropriate visualisation and implement a clear, professional solution.

## Preferred solution

Provide a chart-view selector above the chart with:

* `Comparable growth`
* `Log scale`
* `Absolute rebased level`

The names may be adjusted if clearer.

### Comparable growth view

This should be the default.

The purpose is to compare the shape and relative performance of each series without one extreme compound-growth series making the others unreadable.

Possible approaches include:

* logarithmic cumulative return;
* indexed log-return representation;
* another defensible transformation that preserves relative growth comparison.

The axis label and methodology must clearly state what is being shown.

### Log-scale view

Use a logarithmic y-axis on the rebased index.

Requirements:

* all values must be positive;
* handle invalid or non-positive observations explicitly;
* label the axis clearly;
* retain the common baseline reference;
* ensure hover values remain understandable.

### Absolute rebased-level view

Retain the existing rebased level values for users who want the literal cumulative index.

Requirements:

* clearly warn that extreme long-run compounding can visually compress lower-growth series;
* retain accurate hover values;
* do not use this as the default if it remains unreadable.

## Alternative acceptable solution

A simpler two-mode selector is acceptable:

* `Rebased comparison`
* `Log scale`

provided the default view allows all series to be interpreted meaningfully.

## Additional requirements

* Every series must remain visually distinguishable.
* Users must be able to compare relative performance meaningfully.
* Preserve access to underlying values in hover information.
* Legends, tooltips, reference lines and axis labels must update according to the selected mode.
* The default presentation must optimise for comparison rather than absolute magnitude.
* Ensure colours remain distinct after the redesign.
* Do not solve the problem by placing each company on a separate arbitrary y-axis.
* Do not silently cap, truncate or winsorise genuine values.
* Do not remove Sage solely because it dominates the scale.
* Keep the common baseline date visible.
* Explain the selected transformation in the methodology note or chart caption.
* Retain full historical access through the chart range selector where appropriate.

Add tests confirming that:

* the chart supports the intended view modes;
* the default view is not a single unreadable linear scale dominated by one series;
* all expected company traces are present;
* trace names use company display labels;
* selected axis type and labels update correctly;
* hover data remains valid;
* no series is silently dropped because of scale differences.
